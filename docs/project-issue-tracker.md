# 项目问题与处理状态跟踪

生成日期：2026-07-13

用途：记录当前审阅发现的问题、风险与优化项。以后每处理完一项，就把对应状态从 `未做` 改为 `进行中` / `已完成` / `暂缓`，并补充处理说明。

## 状态说明

| 状态 | 含义 |
|---|---|
| 未做 | 已识别，尚未处理 |
| 进行中 | 正在处理或已有部分改动 |
| 已完成 | 已修复/优化，并通过必要验证 |
| 暂缓 | 暂不处理，保留记录 |

## P0：数据长期增长与保存安全

| ID | 状态 | 问题 | 影响 | 建议处理 |
|---|---|---|---|---|
| DATA-001 | 暂缓 | 主数据和快照都放在 `localStorage`，长期日更可能接近容量上限 | 保存失败、初始化失败、误以为数据已保存 | 暂不做数据体积仪表盘；已有保存失败常驻告警，后续真实需要时再评估体积统计或 IndexedDB |
| DATA-002 | 暂缓 | 本地快照是整份数据复制，主数据越大快照越占空间 | 快照快速吃满 localStorage | 暂不做快照清理/去重入口；保留为长期优化备选 |
| DATA-003 | 暂缓 | 缺少数据体积仪表盘 | 无法判断是哪类数据增长过快 | 暂不作为当前必做；如后续出现容量压力，再显示主数据、快照、各模块条数/大小、风险状态 |
| DATA-004 | 已完成 | `loadData()` 初始化规范化后直接写 `localStorage`，没有捕获写入失败 | 数据大时页面启动可能中断 | 初始化写回已走统一 `persistLocalValue()`，失败时继续渲染并提示导出/清理 |
| DATA-005 | 已完成 | `snapshot-service.js` 快照创建失败后 catch 内可能再次保存失败 | 容量不足时降级逻辑不稳，错误原因不清楚 | 快照写入失败现在返回 `null`，UI 显示失败，不再显示假成功 |
| DATA-006 | 已完成 | 缺少保存失败后的统一恢复引导 | 用户可能不知道下一步该导出还是清理 | 已增加“立即导出 / 管理快照 / 重试保存”的常驻恢复入口 |
| DATA-007 | 已完成 | 主数据写入已有失败提示，但同步状态、配置和快照的 localStorage 写入仍没有统一失败处理 | 容量不足时同步状态可能失真，部分操作仍会静默失败 | 主数据、同步状态、同步配置、转盘同步配置、AI 配置已走统一写入失败提示；快照写入失败已单独返回失败 |
| DATA-008 | 已完成 | 保存失败后没有常驻的“未可靠保存”标记 | 用户关闭提示后容易继续操作，无法判断哪些修改只留在内存 | 侧边栏已增加未可靠保存告警，失败时自动展开并保留到下次成功保存 |

## P1：快照策略与 IndexedDB

| ID | 状态 | 问题 | 影响 | 建议处理 |
|---|---|---|---|---|
| SNAP-001 | 暂缓 | 快照没有区分普通自动快照、手动快照、导入前快照、冲突合并快照 | 重要快照和普通快照价值混在一起 | 快照类型/保护级别不再列为当前必做，后续需要快照治理时再设计 |
| SNAP-002 | 暂缓 | 缺少加星/受保护快照机制 | 重要节点可能在清理时被误删 | 加星/受保护快照机制暂缓，避免在当前阶段扩大范围 |
| SNAP-003 | 暂缓 | 缺少重复 hash 快照去重 | 相同数据状态重复占空间 | 重复 hash 去重暂缓，后续与快照清理策略一起处理 |
| SNAP-004 | 暂缓 | 缺少普通自动快照降采样策略 | 快照数量随使用增长，没有时间层级 | 自动快照降采样暂缓，后续与快照治理统一处理 |
| SNAP-005 | 暂缓 | 缺少快照清理入口 | 用户只能被动等待容量失败 | 基础快照清理入口暂缓；当前优先保证失败不假成功和可导出 |
| SNAP-006 | 暂缓 | 快照仍在 `localStorage`，不适合长期保留 | 与主数据争抢小容量空间 | IndexedDB 迁移属于大迁移，暂缓到体积统计/治理需求更明确后再做 |
| SNAP-007 | 暂缓 | 缺少旧 `localStorage` 快照到 IndexedDB 的迁移流程 | 迁移时可能丢旧快照或重复快照 | 随 IndexedDB 迁移一起暂缓 |

## P2：Cloudflare 同步与云端增长

| ID | 状态 | 问题 | 影响 | 建议处理 |
|---|---|---|---|---|
| SYNC-001 | 未做 | Cloudflare 只能解决远端备份，不能解决本地 `localStorage` 容量 | 本地保存仍可能失败 | 本地快照迁 IndexedDB；Cloudflare 继续保持 JSON 文件同步 |
| SYNC-002 | 未做 | 当前云端同步仍是完整 JSON，长期会越来越大 | 上传、拉取、合并变慢 | 短期保持不变；长期再考虑 current/archive 分文件同步 |
| SYNC-003 | 已完成 | 同步用户名/密码被保存，但底层请求未使用鉴权头 | UI 容易误导；密码明文存储但无实际作用 | 已移除同步配置里的账号/密码字段，并在加载/保存旧配置时清理遗留字段；当前仅保留 Cloudflare Worker 中转地址与路径 |
| SYNC-004 | 已完成 | 自动同步入口较多：启动、定时、页面恢复、延迟保存、手动同步 | 状态管理复杂，慢网络下可能有竞态风险 | 主数据和大转盘同步已增加 pending 队列，同步中再次触发会在当前任务结束后补跑一次 |
| SYNC-005 | 未做 | 主数据未来迁 IndexedDB 后，需要保持 Cloudflare 格式兼容 | 迁移时可能影响现有云端数据 | 迁移主数据时仍上传/下载同一个 `life-plan.json`，不先拆云端结构 |
| SYNC-006 | 未做 | 同步使用 `*.workers.dev` 或独立存储域名时，可能比页面更依赖代理/VPN | 页面可打开但同步不稳定，用户难以判断原因 | 将 Worker 绑定到页面同一自定义域名路径，例如 `/sync/*`，并在设置中给出示例 |
| SYNC-007 | 未做 | 同步是 GET 后 PUT，两个设备在同一时间写入时仍可能跨过本地合并窗口 | 极端并发下后写入者可能覆盖前写入者 | Worker 返回版本号/ETag，PUT 使用条件写入；冲突时重新拉取、合并、再提交 |
| SYNC-008 | 已完成 | 网络请求此前缺少明确超时；错误留痕只需要保留少量关键失败 | 某次请求长时间挂起时同步锁不易恢复，刷新后也难追踪原因 | 已增加 20 秒 `AbortController` 超时、失败原因提示和最近 5 条关键失败留痕；不做完整同步诊断中心 |
